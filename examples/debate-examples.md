# Debate Examples

## 案例 1：创业方向值不值得做

### 议题

`/debate 面向大学生的 AI 复习助手值不值得做成创业项目？`

### 任务分类

- 主分类：`startup`
- 副分类：`planning`
- 理由：核心在判断方向值不值得做，但同时涉及个人投入方式与阶段节奏。

### 参会 Agent

- Paul Graham：判断问题真假与切口是否够窄
- Steve Jobs：判断用户是否会因为核心价值主张而买单
- Munger：检查是否存在创始人投射与错误配置
- Taleb：评估下行风险与是否应该高投入启动

### 分工

- Paul Graham 主做“方向真假”
- Steve Jobs 主做“价值主张是否足够强”
- Munger 主做“认知偏差与机会成本”
- Taleb 主做“失败代价与脆弱性”

### 发言结构

#### Paul Graham
1. 角色职责：判断问题是否真实、切口是否够窄
2. 核心结论：值得验证，但必须缩窄到高压考试场景
3. 判断依据：
   - 复习资料分散是真问题
   - 不是所有课程都强痛点
   - 考试周需求最集中
4. 当前方案最大的一个问题：用户群过宽
5. 我反对的一个常见误判：把“大学生很多”误当“需求很强”
6. 我的具体建议：先只做一个考试场景的手工验证
7. 置信度：中
8. 不确定项：付费意愿与留存频率

#### Steve Jobs
1. 角色职责：判断产品主张是否直接
2. 核心结论：不要卖“AI”，要卖“更快复习到能过考试”
3. 判断依据：
   - 用户不为技术特性买单
   - 价值必须短路径可感知
   - MVP 不能一开始什么都做
4. 当前方案最大的一个问题：主张太宽泛
5. 我反对的一个常见误判：把功能列表当产品价值
6. 我的具体建议：把价值主张压成一句可验证的话
7. 置信度：中
8. 不确定项：不同专业对复习结果的要求差异

#### Munger
1. 角色职责：检查自欺与错误配置
2. 核心结论：最大风险不是技术，而是把自己的学习方式投射成普遍需求
3. 判断依据：
   - 创始人容易高估自己熟悉的问题
   - 学生群体价格敏感
   - 时间成本很可能高于早期信号质量
4. 当前方案最大的一个问题：尚未验证需求就想产品化
5. 我反对的一个常见误判：把兴趣当市场
6. 我的具体建议：先做 20 个真实用户访谈，再决定是否开发
7. 置信度：高
8. 不确定项：校园渠道是否能改变分发效率

#### Taleb
1. 角色职责：识别脆弱性与尾部风险
2. 核心结论：可以试，但不应高投入启动
3. 判断依据：
   - 学生市场低 ARPU 且波动大
   - 考试周期让使用频率不稳定
   - 早期重开发的失败代价不对称
4. 当前方案最大的一个问题：容易在信号不足时就重投入
5. 我反对的一个常见误判：把“MVP”当成大工程借口
6. 我的具体建议：先用表单、社群、人工整理做低成本试验
7. 置信度：高
8. 不确定项：某些高压考试是否存在足够强的刚需

### 主持人汇总

1. 议题重述：判断该方向是否值得启动
2. 参会 Agent 及其职责：如上
3. 共识点：
   - 问题可能真实，但必须切窄
   - 不应高投入启动
   - 价值主张必须直接
4. 核心分歧：需求强度是否足以支撑持续产品化
5. 隐含假设：学生是否会为“更快通过考试”付费或持续使用
6. 真共识：先低成本验证，再决定是否开发
7. 可能受串行影响的结论：无明显串行影响
8. 初步建议：先做低成本验证，不直接开发完整产品
9. 需要审查 Agent 重点检查的地方：验证动作是否足够具体

### 审查意见

1. 本轮讨论总体评分（1-10）：8
2. 履职最好的 Agent：Munger
3. 偷懒或空泛的 Agent：无
4. 缺证据的论点：Jobs 对主张判断较强，但缺少真实用户语料
5. 逻辑跳跃点：无严重跳跃
6. 被忽略的关键问题：分发渠道与付费路径
7. 是否允许进入最终决议：允许
8. 如果不允许，需要补充什么：无

### 最终决议

结论：`值得低成本验证，不值得直接重投入开发`

下一步：
1. 选定 1 个重压考试场景
2. 找 20 个目标用户访谈
3. 先用手工资料整理 + AI 生成做试验

## 案例 2：产品功能该不该加（Quick 模式）

### 议题

`/debate --quick 知识产品要不要加 AI 自动生成关系图谱功能？`

### 任务分类

- 主分类：`product`
- 副分类：`learning`

### 参会 Agent

- Steve Jobs：核心体验裁判
- Zhang Yiming：反馈闭环判断
- Munger：机会成本与误配审查

### 简化发言

#### Steve Jobs
- 核心结论：现在不该加完整图谱，这会让产品失焦。
- 关键依据：用户要的是更快理解，不是更复杂的展示层；图谱是重功能。
- 具体建议：先做轻量“关联建议”而不是完整关系图谱。
- 置信度：中

#### Zhang Yiming
- 核心结论：先做自动反链和推荐闭环，先看用户是否真点击与复用。
- 关键依据：图谱价值取决于反馈回路，不取决于视觉复杂度。
- 具体建议：先上轻量关联模块并埋点验证使用率。
- 置信度：中

#### Munger
- 核心结论：当前最大风险是把“看起来高级”误判成“用户需要”。
- 关键依据：复杂图谱开发成本高，错误优先级会吃掉其他更高价值功能。
- 具体建议：只有当关联建议被高频使用后，才讨论图谱升级。
- 置信度：高

### 最终决议

结论：`不优先加完整图谱，先做轻量关联建议。`

关键风险：可能误判少数深度用户的需求强度。

下一步：
1. 先上轻量关联建议
2. 埋点看点击率与复用率
3. 两周后复盘是否值得升级为图谱

## 案例 3：复杂知识应该怎么学

### 议题

`/debate Transformer 里的 attention 机制应该怎么学，才能既懂原理又能讲给别人听？`

### 任务分类

- 主分类：`learning`
- 副分类：`writing`

### 参会 Agent

- Karpathy：技术直觉与代码路径
- Feynman：概念讲清楚与理解校验
- Ilya Sutskever：为什么它重要

### 主持人汇总

1. 议题重述：不仅要学懂，还要能讲给别人听
2. 参会 Agent 及其职责：如上
3. 共识点：先把 attention 还原成“加权检索”再看矩阵实现
4. 核心分歧：是否一开始就引入大模型级抽象视角
5. 隐含假设：学习目标是应用理解，不是纯论文记忆
6. 真共识：概念、机制、代码、讲述四步都要有
7. 可能受串行影响的结论：Ilya 的抽象判断可能受到前面发言的表达框架影响
8. 初步建议：先用 Feynman 把概念讲明白，再用 Karpathy 对照实现，最后用 Ilya 补“为什么重要”
9. 需要审查 Agent 重点检查的地方：学习输出物是否足够具体

### 审查意见

1. 本轮讨论总体评分（1-10）：8
2. 履职最好的 Agent：Feynman
3. 偷懒或空泛的 Agent：Ilya 略抽象
4. 缺证据的论点：关于“先不看论文也可以”的说法需要学习者目标上下文
5. 逻辑跳跃点：无严重跳跃
6. 被忽略的关键问题：学习产出物形式尚未指定
7. 是否允许进入最终决议：允许
8. 如果不允许，需要补充什么：无

### 最终决议

结论：`按“概念 -> 机制 -> 代码 -> 讲给别人听”的顺序学。`

## 案例 5：审查不通过后的补充讨论

### 议题

`/debate 我是否应该马上 all-in 做一个 AI 创业项目？`

### 首轮问题

首轮讨论中，Naval 和 Jobs 给出了较强方向判断，但：

- Taleb 的风险分析过于简略
- Munger 没有明确指出关键机会成本
- 主持人直接给出“建议尝试”但没有停止条件
- 审查包里没有明确区分事实与推断

### 审查意见

1. 本轮讨论总体评分（1-10）：6
2. 履职最好的 Agent：Taleb
3. 偷懒或空泛的 Agent：Naval
4. 缺证据的论点：关于“现在是窗口期”的论点缺乏具体依据
5. 逻辑跳跃点：从“有机会”直接跳到“应该 all-in”
6. 被忽略的关键问题：学业、现金流、失败代价
7. 是否允许进入最终决议：不允许
8. 如果不允许，需要补充什么：
   - Taleb -> 补完整下行风险与不可逆损失分析
   - Munger -> 补机会成本与错误配置分析
   - 主持人 -> 补事实 / 推断分栏与停止条件

### 补充讨论

#### Taleb 补充
- 最大风险不是项目失败，而是你在没验证前做了不可逆投入。
- 建议：先设置 30 天验证门槛，再决定是否加码。

#### Munger 补充
- 机会成本在于学业底盘和其他更低风险成长路径被你主动砍掉。
- 建议：只有当验证信号明显强于替代选项时，才讨论提高投入。

#### 主持人补充汇总
- 新增事实：当前无稳定现金流，且尚未拿到真实用户付费信号。
- 新增推断：窗口期可能存在，但证据不足以支持“马上 all-in”。
- 更新后的初步建议：保留底线，设阶段门槛，加码前复审。
- 更新后的停止条件：30 天内拿不到有效用户或强验证信号，则不增加投入。

### 二次审查意见

1. 本轮讨论总体评分（1-10）：8
2. 履职最好的 Agent：Munger
3. 偷懒或空泛的 Agent：无
4. 缺证据的论点：仍需用户提供真实现金流数据
5. 逻辑跳跃点：无严重跳跃
6. 被忽略的关键问题：无关键遗漏
7. 是否允许进入最终决议：允许
8. 如果不允许，需要补充什么：无

### 最终决议

结论：`不建议直接 all-in，建议设阶段验证门槛后再决定是否加码。`
