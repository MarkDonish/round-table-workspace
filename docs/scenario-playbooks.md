# Scenario Playbooks

本文件将 Mark 的高频决策场景固化为可复用 playbook。Mark 版优先级不是“哪个场景听起来更厉害”，而是“哪个场景最能带来真实进展”。

默认原则：先控制风险，再低成本验证，再做产品与内容联动，最后再谈放大。

---

## 1. 大学生创业者方向筛选

### 适用场景

- 你有多个方向，不知道先做哪个
- 你在学业、创业、内容之间摇摆
- 你怀疑自己是在追热点，而不是在解决真实问题
- 你担心做了半年，最后既没产品也没成长资产

### 推荐 skill 组合

- 默认：Paul Graham + Munger + Zhang Xuefeng
- 高风险补位：Taleb

岗位分工：
- Paul Graham：判断问题是否真实、切口是否够小
- Munger：找出错误配置、机会成本和自欺点
- Zhang Xuefeng：把选择拉回大学生身份、时间和资源约束
- Taleb（可选）：检查方向是否过度脆弱

### 标准执行步骤

1. 把候选方向写成 2-4 个具体选项，而不是模糊梦想。
2. 用 Paul Graham 判断每个方向是否对应真实用户问题。
3. 用 Munger 判断每个方向的机会成本和自欺风险。
4. 用 Zhang Xuefeng 判断对大学生来说哪一个最现实、最可持续。
5. 如涉及重投入或高机会成本，再用 Taleb 检查下行风险。
6. 给出一个统一建议：先做什么、暂缓什么、放弃什么。

### 一个示例问题

`我现在有三个方向：AI 学习工具、大学生效率内容账号、校园服务类项目，哪个更适合我先做？`

## 2. 低成本验证

### 适用场景

- 想知道一个方向值不值得投时间和钱
- 不想先开发完整产品
- 想先拿到真实用户反馈或付费信号
- 想判断“是继续做，还是尽快止损”

### 推荐 skill 组合

- 默认：Paul Graham + Taleb + Zhang Xuefeng
- 补位：Munger

岗位分工：
- Paul Graham：定义什么信号能证明问题真实存在
- Taleb：控制验证成本和下行暴露
- Zhang Xuefeng：把验证动作拉回现实可执行范围
- Munger（可选）：防止误读信号和自我安慰

### 标准执行步骤

1. 明确你要验证的是哪件事：需求、付费意愿、留存、传播，还是使用频率。
2. 用 Paul Graham 选出最关键的验证信号。
3. 用 Taleb 把验证方式压到最低成本，优先手工、访谈、假门、着陆页、报名页、小样本成交。
4. 用 Zhang Xuefeng 判断这个验证动作是否适合你当前时间、预算和身份。
5. 如你容易把弱信号当强信号，再用 Munger 做认知校准。
6. 输出一份最小验证方案：验证目标、动作、样本、停止条件、升级条件。

### 一个示例问题

`我想做一个 AI 帮大学生整理课程重点的产品，第一步怎么低成本验证，而不是直接开发？`

## 3. 创业机会判断

### 适用场景

- 判断一个创业方向值不值得做
- 识别是否存在真实用户问题
- 比较多个切口哪个更适合先做
- 判断一个项目是伪需求还是真机会

### 推荐 skill 组合

- 默认：Paul Graham + Steve Jobs + Munger
- 高风险补位：Taleb

岗位分工：
- Paul Graham：看问题是否真实、切口是否够窄
- Steve Jobs：看价值主张是否聚焦、产品是否有核心吸引力
- Munger：看是否存在自欺、激励错配和机会成本问题
- Taleb（可选）：看是否暴露于不可逆风险

### 标准执行步骤

1. 定义要判断的机会、用户、场景和预期回报。
2. 用 Paul Graham 判断问题是否真实存在、是否有足够尖锐的切口。
3. 用 Jobs 判断产品主张是否足够集中，是否有“必须存在”的价值。
4. 用 Munger 找出常见误判、忽略的代价和错误激励。
5. 如涉及高成本投入或高不确定性，再用 Taleb 检查极端下行风险。
6. 汇总共识与冲突，给出“做 / 不做 / 缩小后再做”的统一建议。

## 4. 产品功能取舍

### 适用场景

- 决定某个功能该不该做
- 给 MVP 做优先级排序
- 判断一个复杂需求是否应该拆解或直接删除
- 决定某个功能是产品核心还是噪音

### 推荐 skill 组合

- 默认：Steve Jobs + Zhang Yiming + Elon Musk
- 风险补位：Taleb

岗位分工：
- Steve Jobs：判断是否值得存在、是否提升核心体验
- Zhang Yiming：判断是否能形成反馈回路和迭代闭环
- Elon Musk：用第一性原理拆掉多余复杂度
- Taleb（可选）：检查潜在系统风险和不可逆后果

1. 明确功能的目标用户、目标行为和成功指标。
2. 用 Jobs 判断它是否服务产品核心，而不是装饰性需求。

5. 如功能引入较大工程或业务风险，再用 Taleb 做风险校验。
6. 给出“保留 / 砍掉 / 降级 / 延后”的统一建议。

## 5. 内容与产品联动

### 适用场景

- 想用内容获取用户，但不想做纯流量账号
- 想让内容选题反哺产品迭代
- 想把内容当成需求发现、种子用户筛选、转化入口
- 产品和内容都在做，但目前彼此脱节

### 推荐 skill 组合

- 默认：MrBeast + Steve Jobs + Zhang Yiming + Munger

岗位分工：
- MrBeast：判断什么内容最能拿到注意力
- Steve Jobs：判断内容是否强化产品核心主张
- Zhang Yiming：设计内容到产品的反馈与转化机制
- Munger：防止为了流量牺牲用户质量和长期价值

### 标准执行步骤

1. 明确你的产品核心主张，不允许内容和产品各讲各的。
2. 用 MrBeast 设计最容易吸引目标用户的内容切口。
3. 用 Jobs 判断内容是否在强化产品最核心的那一句价值主张。
4. 用 Zhang Yiming 设计内容 -> 私信 / 表单 / 社群 / 测试页 / 产品试用 的转化路径。
5. 用 Munger 检查是否出现“高播放但低价值”的伪增长陷阱。
6. 输出一套联动方案：内容主题池、转化动作、反馈指标、迭代周期。

## 6. 学习复杂概念

### 适用场景

- 理解复杂技术概念
- 学习 AI / 数学 / 物理 / 系统设计
- 看懂代码背后的直觉
- 把抽象概念讲给非专业人听

### 推荐 skill 组合

- 默认：Feynman + Karpathy
- 前沿补位：Ilya Sutskever

岗位分工：
- Feynman：把概念拆成普通人能理解的结构
- Karpathy：补工程直觉、实现感、代码层理解
- Ilya（可选）：补前沿抽象和研究方向感

### 标准执行步骤

1. 明确要学的概念、当前理解程度和最终用途。
2. 用 Feynman 先讲“这是什么、为什么重要、最容易误解什么”。
3. 用 Karpathy 补“系统怎么运作、代码直觉是什么、工程里怎么体现”。
4. 如问题涉及前沿方向或能力边界，再用 Ilya 补充高层视角。
5. 汇总为一套“定义 -> 直觉 -> 例子 -> 常见误区 -> 应用”的统一解释。
6. 最后补一句：这个理解下一步能产出什么。

### 最终输出模板

```markdown

- Karpathy：
- Ilya（可选）：

### 统一解释
- 这是什么：
- 为什么重要：
- 如何工作：
- 常见误区：
- 一个直观例子：

### 下一步输出方向
- 可以写成：
- 可以讲成：
- 可以做成：
```

`Transformer 里的 attention 机制到底是在做什么？`

## 7. 学习转输出

### 适用场景

- 学了一堆东西，但没有形成作品和表达
- 想把学习笔记变成内容、演讲、脚本、产品素材
- 想用输出反过来检验自己有没有真正学懂
- 希望学习、内容、个人品牌形成闭环

### 推荐 skill 组合

- 默认：Feynman + Zhang Xuefeng + MrBeast
- 技术主题可补：Karpathy

岗位分工：
- Feynman：把学习内容整理成可教给别人的结构
- Zhang Xuefeng：把语言拉回普通人听得懂的现实表达
- MrBeast：把输出包装成更有钩子、更容易被点开的形式

## 9. 风险审查与反脆弱评估

### 适用场景

- 审查一个方案的系统性风险
- 判断哪里会在极端情况下失效
- 检查是否存在不可逆损失
- 设计更稳健的行动方案

### 推荐 skill 组合

- 默认：Taleb + Munger
- 技术/机制补位：Zhang Yiming 或 Karpathy

岗位分工：
- Taleb：找尾部风险、脆弱点、不可逆损失
