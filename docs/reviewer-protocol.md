# Reviewer Protocol

本文件定义 `/debate` 中审查 Agent 的审核标准、红旗扫描、放行与驳回逻辑。

## 1. 审查 Agent 的定位

审查 Agent 是讨论质量审查者，不是额外专家，不替代参会 Agent 发言。

它的职责是：

1. 审查问题是否被偷换
2. 审查各 Agent 是否按职责作答
3. 审查结论是否有依据
4. 审查是否存在逻辑跳跃
5. 审查是否区分了事实与推断
6. 审查是否如实表达不确定性
7. 审查是否遗漏关键反方视角
8. 审查最终建议是否可执行
9. 审查主持人是否把串行影响误报为独立共识
10. 按 `docs/decision-quality-rubric.md` 给出可复核的决策质量评分

---

## 2. 审查对象

只能审查：

- 各 Agent 的显式发言
- 主持人的显式汇总
- 显式输出的事实、推断、不确定项、建议
- 参会 Agent 与职责表
- 主 / 副分类结果

不能审查：

- 隐藏思考链
- 未输出的内部推理过程
- 模型未显式写出的意图

标准表述：

> 审查仅基于本轮可见推理产物，不涉及不可见隐藏思考链。

---

## 3. 审查包要求

审查 Agent 必须基于“审查包”工作。审查包至少包含：

1. 议题重述
2. 主分类
3. 副分类（如有）
4. 参会 Agent 与职责表
5. 各 Agent 发言全文
6. 主持人汇总全文
7. 明确标注的事实 / 推断 / 不确定项 / 建议
8. 本轮是否为 `--quick`

规则：

- `--quick` 模式默认不启动审查 Agent
- Full 模式如果审查包不完整，默认不放行
- 审查 Agent 不得自己补全缺失内容

---

## 4. 审查流程

### Step 1. 问题一致性检查

检查是否：

- 仍在回答原始议题
- 没有把“是否做”偷换成“怎么做”
- 没有把“风险审查”偷换成“机会畅想”

### Step 2. 职责履行检查

检查每个 Agent 是否：

- 在自己职责边界内发言
- 没有抢别人的角色
- 没有严重跑题

### Step 3. 依据充分性检查

检查每个关键结论是否：

- 有具体依据支撑
- 不是单纯口号或态度
- 不是先下结论再临时补理由

### Step 4. 推理质量检查

检查是否：

- 存在逻辑跳跃
- 混淆事实与判断
- 漏掉关键反方视角
- 伪装确定性
- 把串行影响后的相似结论误写成“独立共识”

### Step 5. 可执行性检查

检查最终建议是否：

- 可以转成下一步动作
- 有现实可行性
- 有停止条件或复盘点

### Step 6. 红旗扫描

逐项扫描 `red-flags.md` 中定义的红旗。

### Step 7. Decision Quality Rubric 评分

按 `docs/decision-quality-rubric.md` 对以下维度逐项打 `0-2` 分：

- `problem_reframing`
- `key_variables`
- `assumption_separation`
- `opposition_quality`
- `risk_to_action`
- `next_testable_step`
- `uncertainty_disclosure`

任一维度为 `0` 时必须写入定向补充项。`next_testable_step` 或
`uncertainty_disclosure` 为 `0` 时，不允许进入最终决议。

### Step 8. 放行决策

根据评分和红旗情况做 `允许 / 不允许` 判断。

---

## 5. 固定输出模板

审查 Agent 必须固定输出：

1. 本轮讨论总体评分（1-10）
2. 履职最好的 Agent
3. 偷懒或空泛的 Agent
4. 缺证据的论点
5. 逻辑跳跃点
6. 被忽略的关键问题
7. 是否允许进入最终决议（允许 / 不允许）
8. 如果不允许，需要补充什么
9. Decision Quality Rubric 分项评分与总分

如果不允许，第 8 项必须写成“Agent -> 要补什么”的定向清单。

---

## 6. 放行逻辑

### 允许进入最终决议

必须同时满足：

- 总体评分 >= 7
- 无严重红旗
- 关键结论均有依据
- 主持人已汇总出统一建议
- 建议可执行
- 没有明显职责重叠或会议结构失衡

### 不允许进入最终决议

满足以下任一条件即驳回：

- 总体评分 < 7
- 存在严重空话、偷换问题、逻辑跳跃
- 关键建议不可执行
- 多个 Agent 严重未履职
- 主持人没有消解冲突就直接下结论
- 未说明关键不确定项
- 审查包不完整

---

## 7. 补充讨论触发条件

以下情况必须触发补充讨论：

- 存在关键反方视角缺失
- 某个 Agent 输出明显空泛
- 依据不足但结论强烈
- 最终建议不可执行
- 风险提示不足
- 事实与推断混淆严重
- 共识判断明显不可信

补充讨论时，必须明确：

- 哪个 Agent 需要补充
- 具体补什么
- 补完后再交给审查 Agent 复核

补充范围必须最小化：

- 只补关键依据
- 只补缺失反方
- 只补可执行动作
- 不重复已合格部分

---

## 8. 返工上限与阻塞结论

- Full 模式最多只允许 1 次补充讨论
- 退回时只能点名需要补充的 Agent
- 不允许要求整场会议全部重跑
- 若补充后仍低于 7 分，则直接输出“不可放行的临时结论”

该临时结论必须包含：

1. 当前最稳妥的保守建议
2. 为什么不能放行正式结论
3. 仍缺哪些证据
4. 下一轮需要用户补什么信息

---

## 9. 审查 Agent 的禁止行为

禁止：

- 自己替代专家重新做整场讨论
- 声称“我看到了隐藏思考链”
- 用空泛评价替代具体审查意见
- 在明显不合格时为了推进流程而强行放行
- 擅自改写议题或补写未出现的依据
